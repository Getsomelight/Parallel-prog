# Умножение квадратных матриц с использованием CUDA (C++ + Python верификация)

Программа на C++/CUDA умножает две квадратные матрицы на GPU, сохраняет результат в файл и выводит время выполнения.  
Скрипт на Python автоматизирует запуск с разными конфигурациями блоков, проверяет корректность умножения (через `numpy`) и собирает статистику времени для разных размеров матриц.

## Задача

- Реализовать параллельное умножение матриц на GPU с помощью CUDA.
- Перебрать различные конфигурации блоков потоков (8×8, 16×16, 32×32).
- Сохранить результат в файл.
- Измерить время выполнения для размеров матриц 200 – 2000.
- Автоматически верифицировать результат с помощью Python + NumPy.

## Ход работы

1. Написан код на CUDA (`Lab4.cu`):
   - Чтение матриц `matrix_1.txt` и `matrix_2.txt` на хосте.
   - Копирование данных на устройство (GPU).
   - Запуск ядра `matrixMultiplyKernel` с заданными размерами блока (передаются как аргументы командной строки).
   - Измерение времени выполнения ядра с помощью `cudaEvent`.
   - Копирование результата обратно на хост и сохранение в `result.txt`.

2. Для автоматизации экспериментов написан Python-скрипт `solve_verification.py`, который:
   - Генерирует случайные матрицы заданного размера.
   - Запускает `Lab4.exe` с разными размерами блоков (8x8, 16x16, 32x32).
   - Парсит время выполнения из вывода программы.
   - Проверяет корректность умножения через `numpy` (сравнение `np.array_equal`).

3. Эксперименты проведены для размеров матриц  
   200, 400, .., 1800, 2000
   и трёх конфигураций блоков: 8x8, 16x16, 32x32.  
   Для каждого размера выполнено 10 прогонов, усреднённое время занесено в таблицу.

## Выводы

- Использование GPU позволяет получить значительное ускорение по сравнению с CPU-аналогами (для матрицы 2000×2000 время ~0.14–0.28 с против ~30–40 с на CPU).
- Конфигурация блоков 32×32 показывает наилучшую производительность для больших матриц (≥ 1200), благодаря лучшей загрузке мультипроцессоров.
- Для маленьких матриц (200–400) разница между конфигурациями невелика, иногда блоки 16*16 дают минимальное время.
- Увеличение размера блока сверх оптимального (например, 32*32) может не улучшать результат из-за ограничений на количество регистров и разделяемой памяти.
- Верификация через `numpy` подтверждает корректность всех вычислений.
